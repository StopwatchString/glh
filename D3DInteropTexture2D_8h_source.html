2 * @file D3DInteropTexture2D.h
3 * @brief Static functions for creating shared ID3D11Texture2D
4 * objects for use with OpenGL.
5 * @creationdate 2025-02-08
6 * @lastupdate 2025-02-08
7 * @license MIT License (See LICENSE file for details)
8 *
9 * Copyright (c) 2025 Mason Speck
10 *
11 * Description:
12 * A group of static functions and structs to facilitate managing
13 * an ID3D11Texture2D as a mechanism for GPU-level IPC in OpenGL.
14 *
15 * Usage:
16 * Before using these functions, have a valid instance of Direct3DContext.
17 * The definition and handling mechanisms for this are located in
18 * direct3d_utils.h, and this context is used to create and manage
19 * Direct3D resources.
20 *
21 * In the process which will create and fill the texture, create and fill in
22 * a D3DInteropTextureCreateInfo struct. This will be used to generate an
23 * ID3D11Texture2D with the correct parameters. Once this is created, pass
24 * it and the Direct3DContext into createD3DInteropTexture(). This will return
25 * a D3DInteropTexture struct which will contain all necessary handles for
26 * working with this texture as both an OpenGL and Direct3D texture.
27 *
28 * Within the D3DInteropTexture struct, there is a HANDLE
29 * hDxTextureSharedResource. This handle must be shared to the target process
30 * so that it can open the same Direct3D texture resource. NOTE: The target
31 * process must obtain its own copy of the handle, usually through the use
32 * of the DuplicateHandle() function in the Windows API. Simply sharing the
33 * same HANDLE value between processes will not suffice. Typically this is
34 * achieved by sharing the hDxTextureSharedResource and source process id
35 * with the target process, so that it has the tools to create its own handle
36 * via DuplicateHandle().
37 *
38 * Once the target process has obtained the hDxTextureSharedResource, pass it
39 * and its Direct3DContext to openD3DInteropTexture().
40 *
41 * Now both processes have access to the same texture resource, as both an
42 * OpenGL and Direct3D texture. The only stipulation in this interoperability
43 * is that the driver must be given advanced notice of which API you plan to
44 * use before you issue it commands. This is controlled through the interopLock()
45 * and interopUnlock() functions. While 'unlocked' the texture is only
46 * interactable as a Direct3D texture. While 'locked' the texture is only
47 * interactable as an OpenGL texture.
48 *
49 * interopLock() and interopUnlock() **DO NOT** have an impact on inter-process
50 * locking of resources. This is a lock only within the driver to indicate
51 * whether the texture is currently in OpenGL mode. This lock is derived from
52 * OpenGL extensions.
53 */
